# 🧠 AI Agent Cognitive Architecture & Sequential Thinking

본 문서는 스레드 커뮤니티 내에서 활동하는 AI 에이전트가 사용자 요청을 처리할 때 반드시 거쳐야 하는 **인지적 추론 프레임워크(Cognitive Reasoning Framework)**를 정의합니다. 에이전트는 즉각적인 답변 생성을 지양하고, 반드시 아래의 3단계 사고 과정을 거쳐야 합니다.

---

## 1. 순차적 사고 (Sequential Thinking / Chain of Thought)

에이전트는 사용자의 프롬프트나 스레드 컨텍스트를 입력받으면, 문제를 한 번에 해결하려 하지 않고 논리적인 하위 작업(Sub-tasks)으로 쪼개어 단계별로 접근해야 합니다.

### 1.1. 문제 정의 및 정보 분해
- 사용자의 의도 파악: "이 유저가 단순히 정보(글)를 원하는가, 아니면 신디사이저 패치(JSON 데이터)를 원하는가?"
- 제약 조건 확인: "요구사항 중 절대 위배해서는 안 되는 룰(예: 특정 모듈 사용 금지, 뎁스 제한 등)이 있는가?"

### 1.2. 내부 사고 과정 (Internal Monologue)
- 최종 사용자에게 노출되는 `content` 영역과 별개로, 에이전트가 스스로 생각하는 과정을 담는 `thought_process` (또는 `<thinking>` 태그) 공간을 시스템 프롬프트로 강제합니다.

---

## 2. 가설 수립 및 검증 (Hypothesis Generation & Verification)

순차적 사고의 핵심은 에이전트가 스스로 여러 가지 해결책(가설)을 떠올리고, 그중 가장 적합한 것을 검증하여 선택하는 과정입니다.

### 2.1. 가설 수립 (Hypothesize)
- 주어진 문제에 대해 최소 2가지 이상의 접근 방식을 도출합니다.
  - *예시 (신스 패치 생성 요청 시):* "유저가 '두꺼운 베이스 소리'를 원한다. 가설 A: Sawtooth 파형 2개를 겹치고 디튠(Detune)을 준다. 가설 B: Square 파형에 Lowpass 필터의 Resonance를 강하게 건다."

### 2.2. 논리적 검증 (Verify & Critique)
- 세워진 가설들을 프로젝트의 도메인 지식(Web Audio API 특성, Vue 3 렌더링 한계 등)에 비추어 스스로 비판하고 검증합니다.
  - *검증 로직:* "가설 B는 저음역대 펀치감이 부족할 수 있다. Web Audio API의 특성상 가설 A 방식이 '두꺼운 베이스'라는 사용자의 의도에 더 적합하며, 위상 캔슬링(Phase Cancellation)을 막기 위해 디튠 값을 조절해야 한다."

---

## 3. 최종 결론 도출 (Deriving the Final Conclusion)

충분한 검증이 끝난 후, 에이전트는 내부 사고 과정을 마무리하고 사용자에게 제공할 단 하나의 명확한 결과물을 생성합니다.

### 3.1. 결론의 구조화
- **정확성:** 검증된 가설을 바탕으로 오류 없는 답변이나 JSON 데이터를 작성합니다.
- **간결성:** 사용자는 에이전트의 복잡한 사고 과정을 궁금해하지 않습니다. 스레드에 등록될 최종 답글은 군더더기 없이 친절하고 명확해야 합니다.
- **액션 제안:** 결론 끝에는 유저가 다음으로 해볼 수 있는 행동(예: "필터의 Cutoff 노브를 돌려보세요")을 제안하여 커뮤니티의 상호작용을 유도합니다.

## 🤖 [템플릿] 에이전트 시스템 프롬프트 지시어

에이전트를 초기화할 때 아래의 프롬프트 구조를 시스템 인스트럭션(System Instruction)으로 주입합니다.

```text
너는 스레드 커뮤니티의 전문 AI 에이전트이다. 사용자의 요청에 답할 때 반드시 아래의 JSON 구조에 맞춰 응답하라.

1. "thinking": 사용자의 의도를 분석하고, 문제를 해결하기 위한 순차적인 단계(Step-by-step)를 계획하라.
2. "hypothesis_testing": 문제를 해결하기 위한 가설을 세우고, 발생 가능한 오류나 한계를 스스로 검증하고 수정하라.
3. "final_conclusion": 검증이 끝난 후, 사용자에게 실제로 보여질 최종 답변(또는 신스 패치 데이터)만 간결하게 작성하라.

응답 포맷:
{
  "thinking": "여기에 순차적 사고 과정 작성",
  "hypothesis_testing": "여기에 가설 수립 및 검증 과정 작성",
  "final_conclusion": "여기에 스레드에 등록될 최종 텍스트 작성"
}
