# 🔍 Search, Hashtags & Media Upload Guidelines

본 문서는 커뮤니티 내 검색 시스템, 해시태그 기반 분류, 그리고 사진/동영상 파일 업로드 기능 구현 시 준수해야 할 아키텍처 규칙과 보안 가이드라인입니다.

---

## 1. 검색 기능 (Search System)

사용자가 스레드 내용, 작성자, 그리고 공유된 신디사이저 패치를 빠르게 찾을 수 있도록 최적화합니다.

### 1.1. 프론트엔드 (Vue 3) 규칙
- **디바운싱(Debouncing) 필수:** 사용자가 검색창에 타이핑할 때마다 API를 호출하면 서버가 마비됩니다. 입력이 멈추고 최소 `300ms ~ 500ms`가 지난 후에만 백엔드 검색 API를 호출해야 합니다.
- **검색어 길이 제한:** 무의미한 부하를 막기 위해 검색어는 최소 2자 이상, 최대 50자 이하로 제한합니다.

### 1.2. 백엔드 (Node.js) & DB 규칙
- **Full-Text Search (전문 검색) 활용:** 단순 `LIKE '%keyword%'` 쿼리는 테이블 전체를 스캔하므로 성능 낭비가 심합니다. MySQL의 `FULLTEXT INDEX`나 PostgreSQL의 `tsvector`를 활용하여 검색 속도를 높여야 합니다.
- **페이지네이션:** 검색 결과 역시 무한 스크롤을 위한 커서(Cursor) 기반 페이지네이션을 적용하여 한 번에 가져오는 데이터 양(Limit)을 20개 내외로 통제합니다.
- **향후 확장성:** 트래픽이 커지면 DB 부하를 줄이기 위해 Elasticsearch나 Redisearch 도입을 고려할 수 있도록 컨트롤러와 검색 로직을 분리(Service Layer)해 둡니다.

## 2. 해시태그 관리 (Hashtag Management)

해시태그는 스레드와 신디사이저 패치(예: `#Synthwave`, `#Bass`)를 카테고리화하고 트렌드를 파악하는 핵심 수단입니다.

### 2.1. 해시태그 추출 및 정규화
- **백엔드 추출 원칙:** 프론트엔드에서 해시태그 배열을 따로 넘겨받지 않고, 백엔드에서 스레드 `content`를 저장하기 직전에 정규식(Regex)을 돌려 `#` 기호 뒤의 단어를 직접 추출합니다. (보안 및 무결성 유지)
  - *정규식 예시:* `/(?<=#)[^\s#]+/g`
- **정규화 (Normalization):** 대소문자 구분에 따른 중복 생성을 막기 위해, 추출된 모든 해시태그는 소문자로 변환(Lowercase)하여 DB에 저장합니다. (예: `#Synth` -> `synth`)

### 2.2. 데이터베이스 스키마 (M:N 관계)
하나의 스레드가 여러 해시태그를 가지고, 하나의 해시태그가 여러 스레드에 속하므로 다대다(M:N) 관계로 설계합니다.

- **`tags` 테이블:** 고유한 해시태그 사전
  - `id` (PK), `name` (Unique, VARCHAR), `usage_count` (인기 태그 정렬용)
- **`thread_tags` 테이블 (매핑 테이블):**
  - `thread_id` (FK), `tag_id` (FK)
  - *조합 인덱스(Composite Index)* 생성 필수

## 3. 미디어 파일 업로드 (Media & File Upload)

사진, 동영상 등 용량이 큰 파일 처리는 서버 대역폭과 스토리지 비용에 직결되므로 가장 엄격한 규칙이 적용됩니다.

### 3.1. 업로드 아키텍처 (Pre-signed URL 방식 권장)
Node.js 서버가 파일을 직접 받아 스토리지로 전달하면 서버 메모리와 대역폭이 고갈됩니다. **AWS S3 (또는 Cloudflare R2)**의 `Pre-signed URL` 방식을 사용합니다.
1. 프론트엔드가 백엔드에 "나 이 사진 업로드 할게" 하고 요청.
2. 백엔드는 AWS S3에 임시 업로드 허가증(Pre-signed URL)을 발급받아 프론트엔드에 전달.
3. 프론트엔드는 Node.js를 거치지 않고 S3로 파일을 직접 전송.

### 3.2. 용량 및 포맷 제한 (Validation)
- **사진 (Images):** - 최대 용량: 5MB 제한
  - 허용 포맷: `image/jpeg`, `image/png`, `image/webp` (WebP 최우선 권장)
- **동영상 (Videos):**
  - 최대 용량: 50MB 제한
  - 허용 포맷: `video/mp4`, `video/webm`
- **프론트/백엔드 이중 검증:** 프론트엔드에서 용량과 확장자를 1차로 막고, 백엔드에서 Pre-signed URL을 발급할 때 MIME 타입을 2차로 검증해야 합니다.

### 3.3. 파일명 난수화 및 최적화
- **파일명 난수화:** 한글 파일명이나 특수문자로 인한 오류 및 덮어쓰기 방지를 위해, 업로드 시 파일명은 반드시 `UUID`와 타임스탬프를 조합하여 서버가 새로 부여합니다. (예: `f8a1d-20260302.webp`)
- **CDN 서빙:** 업로드된 파일의 원본 URL(S3 주소)을 직접 노출하지 않고, 반드시 CloudFront 같은 CDN을 통해 사용자에게 빠르게 전달되도록 아키텍처를 구성합니다.

### 3.4. DB 설계 (`media_attachments` 테이블)
스레드 본문 테이블과 분리하여 관리합니다.

| Column | Type | Description |
| :--- | :--- | :--- |
| `id` | BIGINT | 미디어 고유 ID |
| `thread_id` | BIGINT (FK) | 첨부된 스레드 ID |
| `file_url` | VARCHAR | CDN을 거친 실제 파일 접근 주소 |
| `media_type` | VARCHAR | `IMAGE` 또는 `VIDEO` |
| `created_at` | TIMESTAMP | 업로드 일시 |
